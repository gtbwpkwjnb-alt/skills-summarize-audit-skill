## ⓪ 加载配置 + 项目记忆

### 第〇步：前置目录检查

审计必须在有明确项目根目录的环境中运行。

| 检查项 | 方法 | 失败处理 |
|:--------|:------|:---------|
| 当前目录是否存在 | `ls` 当前工作目录 | 不存在 → 报错终止，提示用户切换到项目目录 |
| 是否为项目根 | 检查存在以下任一文件：`README.md` / `package.json` / `Cargo.toml` / `pyproject.toml` / `go.mod` / `.git` | 均不存在 → 输出 `⚠️ 当前目录不是项目根，画像可能为空`，询问用户是否继续 |
| 目录深度合理 | 检查路径深度 ≤ 5 层（避免在系统根目录运行） | 路径过深或过浅 → 输出 `⚠️ 目录深度异常`，建议确认 |

检查通过后输出：

```
✅ 前置目录检查: 通过
  目录: {absolute_path}
  项目根: ✅ ({detected_type})
  深度: {N} 层
```

然后读取 `user-profile.md`（含用户高频技能偏好），用于个性化 T2/T3 判定和推荐。检查 `config.yaml` 与 `platforms/*.yaml` 的一致性。

### 第二步：加载项目记忆

读取 `{project_root}/memories/MEMORY.md`：

- **文件存在** → Read 全量内容，提取各区块合并到项目角色画像
- **文件不存在**（新项目）→ 输出交互问卷向用户了解：项目用途/定位、技术栈、兴趣领域、关键路径或入口文件。收到回复后写入初始 `memories/MEMORY.md`

### 第三步：模式检测与确认

1. 检查当前调用意图是否明确为审计/审查需求。若用户请求与审计无关则**终止流程并提示用户正确使用方式**。
2. 若确为审计调用，设置模式标志：
   - 检查触发词是否含 `--ci` 或子命令 `ci` → 设置 `$CI_MODE=true`
   - CI 模式特征：无交互（跳过⑦用户确认，自动执行 AUTO 操作）、JSON 输出、跳过 ④-bis、max_tokens 预算限制
   - CI 模式 Token 预算：`max_tokens` 取自 `config.yaml` 中 `ci.max_tokens`（默认 8000），超限时自动降级步骤（优先跳过 ④-bis → ⑤a → ⑤b）
3. 设置 `$AUDIT_MODE`：
   - `"skill"` → 后续全量执行所有步骤
   - `"project"` → 跳过步骤④-bis/描述优化，聚焦画像+环境+工具链

### 第四步：并行化编排（v5.8.0 更新）

```text
时序: ⓪ → (①+③) → ② → ②-bis → ⑤a → ④ → (④-bis+⑤b) → ⑥ → ⑥-bis → ⑦ → ⑧
阶段: 前序 → 并行2 → 串行3 → 串行4 → 串行5 → 串行6 → 并行7 → 串行8
```

| 阶段 | 步骤 | 说明 |
|:------|:------|:------|
| **1 前序** | ⓪ 配置+记忆+模式 | 必须最先完成，模式决定分支（含 CI 模式检测） |
| **2 并行** | ① 画像 + ③ 参考数据 | 互不依赖 |
| **3 串行** | ② 已安装技能清单 | 输出技能名列表+容量数据，供⑤a和②-bis使用 |
| **4 串行** | ②-bis 容量分析 | 依赖②的容量数据；计算灵活承载上限 |
| **5 串行** | ⑤a 外部信号获取 | 依赖②的技能名列表；获取后回填 Community 分 |
| **6 串行** | ④ 评分+冗余+健康 | 依赖②和⑤a结果 |
| **7 并行** | ④-bis 深读 + ⑤b 推荐 | ④-bis 依赖④评分结果做分级（S/A全量·B轻量·C/D跳过），⑤b 依赖画像+④结果；两者互不依赖，可并行 |
| **8 串行** | ⑥ 报告 → ⑥-bis 验证 | 依赖全部；CI 模式跳过 ④-bis 相关输出 |
| **9 串行** | ⑦ 执行 → ⑧ 持久化 | 依赖验证结果；CI 模式自动执行 AUTO 操作 |

> **⑤a 阻塞策略**：⑤a 最多等待 2 次重试（每次 10s 超时）。若全部失败：
> - Community 维度标注「⏸ 信号待获取」，跳过该维度评分
> - 综合分用剩余 4 维重算权重（Fit 43.75% / Value 25% / Fresh 18.75% / ROI 12.5%）
> - **禁止**使用伪造默认值（如固定25分）填充 Community 维度
> - 报告中明确标注「离线模式 · 外部信号不可用」

## ① 项目画像

只扫描当前工作目录（或用户指定的 `--path`），不混入 `scan_paths` 的技能安装目录。

### 第一步：目录基础信息
- 项目名称（从目录名或 `README.md` 首行 / `package.json` `name` 字段）
- 项目根路径
- 目录结构概览（`ls` 第一层）

### 第二步：关键文件深度分析
Read 以下文件（存在则读，不存在跳过）：`README.md`、`AGENTS.md` / `MEMORY.md`、`package.json` / `pyproject.toml` / `Cargo.toml` / `go.mod`、`.gitignore`

### 第三步：文件特征统计（尊重 `.gitignore`）
语言文件分布、配置文件、测试文件、技能相关、脚本工具

### 第四步：构建多维项目角色画像

**复合类型识别（v5.7.0新增）**：

1. 按主特征定类型（如 package.json 命中 React → "Web前端-React"）
2. 继续检测其他特征文件（如同时存在 pyproject.toml → "Python"）
3. 若命中 ≥2 个独立类型 → 生成复合类型：`Web全栈-React+Python`
4. 复合类型的推荐技能列表 = 各子类型推荐技能取并集+去重（priority 取最高值）
5. 画像中记录：`类型: Web全栈-React+Python（复合: Web前端-React + Python）`

```text
┌─ 项目画像 ──────────────────────────────┐
│ 目录 {path}  类型 {从文件+project-types} │
│ {若复合: 复合: Web全栈-React+Python}     │
│ 用途 {一句话}  活动 {5-8项核心活动}       │
│ 技术栈 {语言+框架+工具}  兴趣 {领域}     │
│ 痛点 {数据驱动}  入口 {关键路径}         │
└─────────────────────────────────────────┘
```

### 痛点推导规则（数据驱动，禁止 agent 猜测）

| 数据源 | 导出痛点 |
|:--------|:---------|
| 核心活动无对应技能覆盖 | "技能覆盖率低: 核心活动{活动}无对应技能" |
| 技能分布集中单一领域 | "技能使用偏科: 集中在{领域}" |
| 大量 T3 仍已安装 | "技能库臃肿: {N}个T3占用扫描路径" |
| config scan_paths 指向不存在目录 | "配置失效: {路径}不存在" |
| 技能依赖的外部工具缺失 | "依赖断裂: {技能}依赖{工具}已缺失" |
| 同类工具多版本共存 | "版本碎片: 多版本共存" |

> 无任何数据源命中时，痛点字段输出空行。

### 第五步：画像强制持久化

1. 确保 `{project}/.agents/` 目录存在
2. 将画像写入 `{project}/.agents/project-profile.md`
3. 设置 `$PROFILE_READY=true`
4. 步骤④评分前检查 `$PROFILE_READY`
5. 持久化失败 → 输出警告但降级为内存画像继续

| 采集项 | 采集方式 | 用途 |

## ②-bis 容量分析引擎（v5.8.0 新增）

> 核心诊断能力：分析当前 agent 环境下每个已安装技能/工具的 token 成本，计算灵活承载上限。

### 计算公式

```
灵活承载上限 = (上下文可用空间 - 固定消耗) / 单 skill 平均注入成本

上下文可用空间 = Agent 实际可用 Token（非原生窗口）
固定消耗 = SystemPrompt + 对话历史留存 + 用户输入预留
单 skill 平均注入成本 = 取决于 skill 加载机制
```

### 各 Agent 计算模型

| Agent | 上限约束 | skill 成本计算 | 固定消耗估算 |
|:---|:---|:---|:---|
| **CodeBuddy** | 16K 硬限制 | 按 use_skill 调用叠加 | SystemPrompt 2800 + 历史 3000 + 用户输入 500 = 6300t |
| **Claude Code** | skillListingBudget=1%×200K=2000t | 全部 skill 描述计入 listing budget | SystemPrompt 3000 + 历史 8000 + 用户输入 1000 = 12000t |
| **Cursor** | 动态，无固定上限 | 按需加载，仅计当前 | SystemPrompt 2500 + 历史 3000 + 用户输入 500 = 6000t |
| **Codex CLI** | 约 60K-100K | 类似 Claude Code 机制 | SystemPrompt 3000 + 历史 6000 + 用户输入 1000 = 10000t |
| **Windsurf** | 约 50K-80K，MCP≤100工具 | 类似 Cursor 动态选择 | SystemPrompt 2500 + 历史 4000 + 用户输入 500 = 7000t |
| **WorkBuddy** | 取决于订阅/模型 | 基于 OpenClaw 架构 | SystemPrompt 2500 + 历史 3000 + 用户输入 500 = 6000t |

### 容量分级判定

```
剩余空间 / 单 skill 平均成本 = 可再装数量

可再装 ≥ 5  → 🟢 充裕，放心扩展
可再装 2-4  → 🟡 注意，选择性扩展
可再装 1    → 🟠 临界，仅可装轻量 skill（<2000t）
可再装 0    → 🔴 已满，必须精简现有 skill 才能扩展
```

### 输出格式

输出到步骤⑥报告的 `capacity_analysis` 块，JSON schema 见 `references/ci-output-schema.md`。

## ④ 三级分层 + 七维评分

### 第一步：三级分层

| 层级 | 规则 | 动作 |
|:------|:------|:------|
| **T1 核心** | 技能的主要用途直接命中"核心活动"中的某一项 | 保留（自动） |
| **T2 通用** | 技能不局限于特定项目类型，当前项目可用 | 保留 |
| **T3 专业** | 技能为特定项目类型设计，与"核心活动"零交集 | 建议归档 |

### 第二步：七维评分（五档制 S/A/B/C/D + 强制理由）

**档位定义：**

| 档位 | 分数 | 含义 | 后续动作 |
|:---:|:---:|:---|:---|
| **S** | 5 | **核心必备** — 直接命中 ≥2 项核心活动 | ✅ 保留 |
| **A** | 4 | **高度适配** — 直接命中 1 项核心活动 | ✅ 保留 |
| **B** | 3 | **可用** — T2 通用技能 | 🟡 监控 |
| **C** | 2 | **边缘** — T3 领域技能 | ❌ 建议归档 |
| **D** | 1 | **建议移除** — 零交集 + 零调用 | ❌ 强制归档 |

**七维标准（v5.8.0）：**

| 维度 | 权重 | S(5) | A(4) | B(3) | C(2) | D(1) |
|:---|:---:|:---|:---|:---|:---|:---|
| **Fit** 项目匹配 | 30% | 命中 ≥2 核心活动 | 命中 1 核心活动 | T2 通用 | T3 领域 | 零交集 |
| **Value** 预期价值 | 20% | 命中≥2核心活动+高频场景 | 命中1核心活动 | T2通用，当前项目可用 | T3领域，低频使用 | 与画像零关联 |
| **Fresh** 版本时效 | 17% | 版本 ≥ registry | 落后 ≤1 小版本 | 无版本号但可用 | 落后 ≥1 大版本 | 严重过时 |
| **Community** 外部信号 | 18% | ⭐≥100 或近1月更新 | ⭐≥10 或近6月更新 | 本地/无公开repo | 信号弱/未知 | 明确废弃 |
| **ROI** 成本效益 | 10% | >+3000t/run | +1000~3000t/run | 0~1000t/run | -500~0t/run | <-500t/run |
| **Novelty** 新颖性 | 10% | 独特能力，无替代品 | 有差异化，竞品仅部分覆盖 | 标准实现，1-2竞品重叠 | ≥3竞品高度重叠，无差异化 | 完全冗余，可被已有替代 |
| **Contamination** 污染 | 5% | 纯目标语言，无混合 | ≤10%混合，不影响理解 | 10-30%混合，轻微影响 | >30%混合/触发词不一致 | 严重混合/触发失效 |

**Novelty 检测方法（v5.8.0）：**
1. WebSearch `"{skill_name} alternative AI agent skill"`
2. 对比已安装技能 + skill-registry.yaml 功能描述
3. 与竞品数据库（skill-validator / SkillsMP）交叉比对
4. 仅 S/A 档执行（④-bis 分级策略），CI 模式跳过

**Contamination 检测方法（v5.8.0）：**
1. 正则检测中英文字符比例
2. 触发词矩阵一致性校验（SKILL.md ↔ platforms/*.yaml）
3. 描述质量检查时同步执行
4. 本地计算，几乎零 token 成本

**安全子项（v5.8.0，融入 Value 维度）：**
- 技能通过 SkillSpector 扫描 → Value +0.5 档
- 技能含 Critical 漏洞 → Value 自动降为 D
- 技能含 High 漏洞 → Value 自动降为 C

**评分执行规则：**
1. 逐维判定 → 选 S/A/B/C/D
2. 计算综合分：`综合 = Fit×0.30 + Value×0.20 + Fresh×0.17 + Community×0.18 + ROI×0.10 + Novelty×0.10 + Contamination×0.05`（S=5,A=4,B=3,C=2,D=1）
3. 映射最终档位：≥4.0→S · ≥3.0→A · ≥2.0→B · ≥1.5→C · <1.5→D
4. **强制理由**：每个技能 1 句理由，无理由=未评分

**CI 模式降维**：`$CI_MODE=true` 时自动降为 5 维（跳过 Novelty + Contamination），权重重新分配为 Fit 35% / Value 25% / Fresh 20% / Community 15% / ROI 5%。

输出格式：
```
#N  {技能名}  {档位}/{综合分}  Fit={档位}·Value={档位}·Fresh={档位}·Comm={档位}·ROI={档位}·Novelty={档位}·Contam={档位}
    理由: {1句话说明最关键的评分依据}
```

### 第三步：冗余检测 + 健康管理

**① 功能重叠检测**
- 技能-技能重叠：关键词重叠度 >60% → ⚠️ 建议二选一
- 技能-MCP 跨层重叠：2+ 关键词命中 → ⚠️ 标注
- 互补关系（方法型+执行型）→ ✅互补

**② 价值四象限矩阵**：高匹配高价值✅ 高匹配低价值🔶 低匹配高价值🟢 低匹配低价值🔴

**③ 容量预警**：
- 总技能数 > `max_total_skills`（默认30）→ 🚨
- T3 占比 > `max_t3_ratio`（默认30%）→ 🚨
- 与画像零匹配技能>0 → 🟡

**④ 依赖链检查**：对每个用户技能，检查其引用的外部命令/文件是否存在

## ④-bis 逐技能深度阅读与分析

**分级执行（v5.7.0新增）**：根据④评分结果分级，避免对低价值技能消耗过多 token。

| 档位 | 策略 | 操作 | token预算/技能 |
|:---:|:---|:---|:---:|
| **S/A** | 全量深读 | 读全量+六维分析+维护清单+外部验证≥2来源+版本回写 | ~2,000t |
| **B** | 轻量检查 | 读 frontmatter+首段+精简维护清单+1次外部验证 | ~800t |
| **C/D** | 跳过 | 仅输出归档建议，不做深度阅读 | ~100t |

对每个已安装用户技能（editable=true），按档位执行：

1. **读内容**（按档位）：S/A读全量 · B读frontmatter+首段 · C/D跳过
2. **六维分析**（仅S/A）：作用·适配·缺口·建议·维护·参考（模板见 `references/deep-analysis-template.md`）
3. **维护清单**（S/A完整 · B精简）：版本/环境/描述/注意
4. **外部参考验证**（S/A ≥2来源 · B 1来源）：agent-reach/web_fetch
5. **自指校验**（仅S/A）：检查 skills-audit 自身的深度分析是否被应用
6. **版本自动回写**（S/A/B）：对比从 SKILL.md 提取的版本号与 `skill-registry.yaml` 中记录：
   - 一致 → 跳过
   - 不一致 → 更新 `skill-registry.yaml` 中对应条目的 `version` 字段
   - registry 中无此技能 → 追加新条目（含 name/version/tags/description）

> 缺口必须经工具验证（ls/Test-Path/grep）。网络不可用时标注「⏸ 离线」。
> 子命令 `轻量`/`quick` 跳过全部 ④-bis。项目审计模式自动跳过。
> 版本自动回写解决 registry 手动维护滞后问题，保持 Fresh 维度评分准确性。

## ⑤b 三层推荐引擎（画像驱动 · 置信度量化）

> 黄金法则：每一条推荐都必须附带信心指数（0-10）和可追溯的证据链。信心<5 自动折叠。

**置信度评分规则：**

| 维度 | 权重 | 计算方式 |
|:------|:------|:---------|
| **匹配度** — 语义覆盖 | 40% | 活动交集数 / 画像总活动数 × 10 |
| **数据源** — 可靠等级 | 30% | project-types.yaml→9；skill-registry→7；Web≥2来源→6；Web单来源→4；推测→2 |
| **交叉验证** — 一致性 | 20% | ≥3来源→10；2来源→8；单一→5；矛盾→0 |
| **时效性** — 更新时间 | 10% | 30天→10；6月→8；1年→6；>1年→3；未知→5 |

**综合信心指数 = 匹配度×0.4 + 数据源×0.3 + 交叉验证×0.2 + 时效性×0.1**

**ROI 预估（每项必算，v5.7.0精化）：**

> ROI = 预期节约 - 加载成本
> - 加载成本 = 基于 SKILL.md 行数估算：<50行→~500t · 50-200行→~1,200t · >200行→~2,500t
> - 预期节约 = 基于画像推断的每次调用节约 token 数

| 画像场景 | 节约 tokens/run |
|:---------|:-----------|
| 高频任务(调试/搜索/总结) | +1,000~3,000 |
| 中频任务(审查/验证/撰写) | +300~1,000 |
| 低频/领域专用工具 | +100~300 |
| 与画像零交集 | 净消耗 -500 |
| T3+零交集 | 净消耗 -1,000 |

**ROI档位映射（5档连续制）：**

| 档位 | 净收益范围 | 说明 |
|:---:|:---|:---|
| S | >+3,000t/run | 极高效 — 高回报高频场景 |
| A | +1,000~3,000 | 高效 — 明确正向收益 |
| B | 0~1,000 | 中性偏正 — 可保留 |
| C | -500~0 | 轻度消耗 — 需关注 |
| D | <-500 | 净消耗 — 建议归档 |

### A层：内置映射匹配

从 `references/project-types.yaml` 匹配当前画像.类型，计算匹配度和置信度。

输出格式：
```
{等级} {技能名}  信心:{分}/10  ROI:{分}/run
  📌 命中: {活动覆盖说明}
  📊 交叉验证: {来源说明}
  💡 预期效果: {...}
  ⚠️ 注意: {...}
```

### B层：Gap 分析

从画像.活动+画像.兴趣提取活动关键词，与已安装技能做交集。未覆盖活动→评估相关性+决策建议。

### B-bis：自定义规则（v5.7.0新增）

若 `config.yaml` 中 `custom_rules.enabled = true`，逐条匹配规则：

1. 读取画像字段（type/tech_stack/activity_count）
2. 按 condition 匹配（field + contains/equals/less_than）
3. 执行 action：
   - `add_skill` → 追加推荐（标记来源: "自定义规则"）
   - `remove_skill` → 从推荐列表移除（记录原因）
   - `suggest` → 在报告中追加建议信息
4. 自定义规则优先级高于 project-types.yaml 默认推荐

### C层：网络搜索补充

中英文各1次搜索（各取前5条）。同名/同功能去重，与已安装冲突时对比信心指数。

### 去重合并（v5.7.0新增，A+B+C层汇总后执行）

1. 按技能名去重：A层 + B层 + C层 → 合并列表
2. 同名技能优先规则：
   - A层 vs C层 → 保留A层（内置映射可靠性高于网络搜索）
   - B层 vs C层 → 保留B层（Gap分析基于画像活动，更精准）
   - A层 vs B层 → 保留A层（内置映射优先级更高）
3. 信心指数：保留原始来源的信心值，不做合并加权
4. 最终推荐列表按信心降序排列

### 推荐终选规则

- 信心≥7 → ⭐⭐⭐⭐ 强烈推荐
- 信心5-6.9 → ⭐⭐⭐ 建议评估
- 信心<5 → ⭐⭐ 仅供参考（移入不推荐区）
- 已安装技能不进推荐表

### 不推荐清单规则

| 来源 | 判定条件 |
|:------|:---------|
| 信心<5 | 4维度加权平均<5 |
| ROI<−1,000 | 净消耗过大 |
| 已安装T3且<50分 | 零交集+低分 |
| 已安装T3+与画像零匹配 | 零交集+零价值 |
| 与核心技能重叠>70% | 功能重叠 |

## ⑥-bis 审计结果验证

**第一步：数据一致性检查**

| 检查项 | 方法 | 失败处理 |
|:--------|:------|:---------|
| 评分表与画像一致 | 确认Fit最高分技能命中核心活动 | ⚠️"评分-画像偏差" |
| Community回填状态 | 确认缓存已写入 | "部分信号未获取" |
| 评分理由完整性 | 每个技能附带1句理由 | ⚠️"评分理由缺失"，阻塞执行 |
| 推荐与不推荐无冲突 | 同技能不同时出现在两清单 | 立即更正 |
| 去重完整性 | 评分表中无同名技能 | ⚠️"去重遗漏" |
| **评分-决策一致性（v5.7.0新增）** | 对比本次评分与用户上次操作（归档/保留/安装）的一致性 | 不一致项>30%→记录到 stats.json 供权重自进化 |

**第二步：抽样交叉验证**：随机1个T1+1个T3技能验证评分合理性

**第三步：验证结果输出**：

**评分-决策一致性自进化（v5.7.0新增）**：

1. 从 `self-audit-issues.json` 读取用户上次审计的决策（归档了哪些、安装了哪些）
2. 对比本次评分：若用户归档了高评分技能（≥A），说明评分维度可能偏差
3. 记录偏差到 `stats.json.score_deviation`：`{dimension: "Fit", delta: +0.02}`（暂不自动调整，积累 ≥5 次审计后触发权重微调建议）
4. 在报告中标注：`📊 评分校准: 基于{5}次审计数据，建议微调 Fit 权重 35%→{33%}（需用户确认）`
```
✅ 审计验证: 通过
  - 数据一致性: 4/4 项检查通过
  - 抽样验证: T1 合理 · T3 合理
  - 报告可信度: 高
```

## ⑦ 执行（用户确认后）

操作范围：仅用户层技能（editable=true）。系统/插件层技能只读。

- **归档**：复制到 `archive_dir`，原路径重命名为 `{原名}.archived`
- **修改描述**：创建 `.description.bak` 备份 → Edit SKILL.md frontmatter description
- **安装**：给出安装命令示例
- **更新**：给出更新方式
- **回滚**：报告末尾附回滚命令

执行前确认模板：
```
将执行 {N} 项操作：
  自动 {A} 项: {list}
  需确认 {K} 项: {list}
  依赖暂停 {S} 项: {list}
回复「执行」继续全部，或指定编号如「1 3」
```

**审计反馈收集（v5.7.0新增）**：所有操作执行完成后，输出简短反馈问卷：

```
📊 本次审计效果如何？
  [1] 有帮助，会按建议执行
  [2] 部分有用，有些建议不适用
  [3] 帮助不大，评分/推荐不准确
  [4] 跳过反馈
```

反馈写入 `stats.json.user_feedback` 数组，格式：`{timestamp, rating: 1|2|3, skipped: false}`。
反馈数据用于：
- rating=1/2 占比 >80% → 当前评分权重有效
- rating=3 占比 >30% → 触发评分权重自进化检查
- 连续3次 rating=3 → 在报告中建议用户调整 custom_rules

## ⑧ 持久化（始终执行）

1. 项目画像 → 双写：`{project}/.agents/project-profile.md` + `project-profiles.json`
2. 运行统计 → `stats.json`（含最近3次审计的关键指标快照：均分/活跃数/T3占比/冗余数）
3. 审计日志 → `activity-log.jsonl`（保留最近50条）
4. 外部信号缓存 → `external-signals-cache.json`
5. **缓存GC（v5.7.0新增）**：清理 `external-signals-cache.json` 中不在当前技能列表中的条目，防止已卸载技能缓存膨胀
6. 审计快照 → 从 `activity-log.jsonl` 倒数第2条读取
7. 自检缺陷追踪 → `self-audit-issues.json`

**趋势数据提取（v5.7.0）**：从 `stats.json` 读取最近3次审计的 `avg_score`/`active_count`/`redundant_count`，生成趋势行。首次审计或不足3次时显示「首次审计·趋势将在下次显示」。

持久化失败不阻塞主流程。

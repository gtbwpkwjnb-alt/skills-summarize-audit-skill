# ⑥ 生成报告

报告生成阶段默认只读。读取 `references/output-contract.md`，输出后不写入 `.data/stats.json`；用户确认保存审计历史后才在⑧持久化。

**30秒摘要块**（始终最先输出，v5.9.1 趋势行）：
```text
审计状态: partial
证据覆盖: observed <N> / inferred <N> / estimated <N> / unavailable <N>
健康度: <状态> · 证据: <路径或命令> · 限制: <缺失数据>
操作建议: [需确认] <N> 项；本轮未写入文件
未获取数据: <外部信号/质量账本/趋势历史等>
```

**容量报告块**：agent 类型、固定消耗、token 成本表、容量判定。

**评分表趋势列**（v5.9.1）：仅存在已确认保存的两次以上快照时显示 ↑/↓/→；否则输出 `unavailable`。

**CI 模式**：JSON 输出，含 summary/skills/recommendations/alerts/capacity_analysis/exit_code/token_usage。

**描述质量检查**：语言自适应→格式检查→精炼→翻译→双语拼接。默认输出候选修改，不直接写入任何 description。

**生态雷达区块**（v6.0.0 — 条件输出，仅在发现能力缺口+社区替代品时显示）：

```
🌐 生态雷达 — 工具优化建议

┌──────────────────────────────────────────────────────────────────┐
│ 工具             质量    社区替代              建议              │
├──────────────────────────────────────────────────────────────────┤
│ <当前工具>    <observed>  <候选与证据URL>       [需确认] 建议     │
│ <能力维度>    <状态>      <许可证/兼容性>        <作用域>         │
│ 限制: 未经用户同意的社区搜索或缺失证据不得填充本区块             │
└──────────────────────────────────────────────────────────────────┘
```

触发条件（需同时满足）：
1. Phase ④ 质量信号发现至少 1 个工具的某能力维度评分 < quality_threshold
2. Phase ⑤-aa 社区 Feed 搜索到同一能力维度的替代品
3. 替代品的该能力评分 ≥ 当前工具 +2

每个推荐附带：
- 📌 问题: 当前工具的哪个能力薄弱 + 证据（错误次数/质量评分）
- 🎯 方案: 社区替代品 + 置信度
- 🔗 互补性分析: 新旧工具的能力重叠与互补关系
- 💡 建议: 安装/替换/并存的决策建议
- 📁 作用域: `project`/`global`/`plugin`/`defer`、目标路径与理由
- 🧾 对比证据: 候选 URL、许可证、活跃度、兼容性与安全备注
- 🔎 事实状态: 每项结论的 observed/inferred/estimated/unavailable 标记

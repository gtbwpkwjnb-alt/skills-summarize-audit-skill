# Skills Audit v5.3.0 — MCP 跨层扫描 · 能力地图 · 重叠检测

> **六大核心能力**: ①项目记忆驱动精准画像 ②五维评分+冗余检测+健康管理 ③逐技能深度阅读+维护清单 ④先搜索后开发方法论 ⑤双模式（技能审计/项目审计）⑥条件输出报告（按需组合，告别空块）
>
> **核心原则**: ⚠️ 黄金法则 — 先搜索验证，再决策/开发。涉及技能开发/修改/推荐时，必须先走 agent-reach / web_fetch 检索需求文档、开发文档、最新技术动态，引用 ≥2 个独立来源。禁止在未验证的情况下盲目推荐或编写技术方案。
>
> **记忆驱动**: 画像即锚点。4层金字塔：L0 `~/.agents/AGENTS.md`（hook自动注入全局底线）> L1 `AGENTS.md`（项目钩子）> L2 `memories/MEMORY.md`（事实库，审计自动读写）。项目画像持久化到 `{project}/.agents/project-profile.md`，全局人物画像在 `~/.agents/skills/skills-audit/user-profile.md`，两者头部标注关联路径确保 agent 识别读取链路。

## 触发方式

**ZCode**: `技能审查` / `skills-audit`（技能审计模式）· `项目审查` / `项目审计` / `project-audit`（项目审计模式）
**Claude Code**: `/skills-audit` · **Codex/Cursor**: `技能审查` / `skills-audit`

**参数**: 项目路径可选。不传则用当前工作目录。
**触发规则**: `技能审查`/`skills-audit` 触发技能审计模式；`项目审查`/`项目审计`/`project-audit` 触发项目审计模式。触发后由步骤⓪-第三步的模式确认分流至对应分支。

## When to Use / 何时使用

- **技能审计模式**：定期维护技能库健康度（建议每 2-4 周一次），或在技能库发生较大变更后（安装/卸载/升级后）
- **项目审计模式**：接手新项目时快速了解技术栈和工具链现状，或在项目重构前评估环境健康度
- **推荐场景**：多技能工作区健康检查、项目管理规范度提升、新人入职环境熟悉

## When NOT to Use / 何时不使用

- **单次文件编辑**：不要用本技能做单文件的代码审查或修改（那是 code-review 或 summarize 的职责）
- **实时调试**：本技能不诊断运行时错误或调试代码
- **极简项目**：无任何技能依赖的纯代码项目不需要审计（审计本身会增加 token 开销）
- **仅需安装单技能**：如果目标只是安装某个技能，直接使用 skill-installer 更高效
- **低资源环境**：审计消耗较多 token（约 5K-15K tokens/次），低 token 预算环境下慎重使用

## 执行流程

### ⓪ 加载配置 + 项目记忆

**第一步：加载配置**
读取同目录下 `config.yaml`，确定：
- `scan_paths` — 要扫描的技能/工具存放目录
- `project_scan` — 项目扫描配置（忽略模式、最大深度等）
- `version_check` — 版本检查源 URL
- `data_dir` — 持久数据目录
- `external_signals_cache` — 外部信号缓存 TTL 配置
- `memory_config` — 项目记忆文件配置
- `dev_search_policy` — 先搜索后开发策略参数
- `health_thresholds` — 健康阈值

然后读取 `user-profile.md`（含用户高频技能偏好），用于个性化 T2/T3 判定和推荐。
检查 `config.yaml` 与 `platforms/*.yaml` 的一致性，不一致则输出警告。

**第二步：加载项目记忆**
读取 `{project_root}/memories/MEMORY.md`：

- **文件存在** → Read 全量内容，提取 `<environment>`、`<tech_stack>`、`<interests>`、`<paths>`、`<project>`、`<commands>`、`<architecture>` 各区块，合并到项目角色画像
- **文件不存在**（新项目）→ 输出交互问卷向用户了解：
  1. 项目用途/定位一句话描述
  2. 使用的技术栈（语言/框架/运行时）
  3. 兴趣领域（如游戏/炒股/写作/玄学/数据科学）
  4. 关键路径或入口文件
  收到回复后写入初始 `memories/MEMORY.md`

**第三步：模式检测与确认**

> BM25 检索可能因句子包含触发词而误触发（如"上次技能审查的结果"含"技能审查"但非意图调用），**此步负责过滤误触发**。

1. 检查当前调用意图是否明确为审计/审查需求。若用户请求与审计无关（如"帮我看看上次技能审查的结果"并非再次执行审计），则**终止流程并提示用户正确使用方式**。
2. 若确为审计调用，设置 `$AUDIT_MODE`：
- `"skill"` → 后续全量执行所有步骤
- `"project"` → 跳过步骤④-bis/描述优化，聚焦画像+环境+工具链

> 模式检测后立即读取 references 文件（project-types.yaml / skill-registry.yaml）。$AUDIT_MODE="project" 时跳过④-bis 和描述优化。

**第四步：并行化编排（v5.2.0 新增 — 减少串行等待）**

> 将无关步骤解耦为阶段化并行执行，大幅减少 agent tool call 轮次。

```text
时序: ⓪ → (①+③) → (②+⑤a) → ④ → (④-bis+⑤b) → ⑥ → ⑥-bis → ⑦ → ⑧
阶段: 前序 → 并行2 + 并行3 → 串行4 → 并行5 → 串行6 → 串行7
```

| 阶段 | 步骤 | 说明 |
|------|------|------|
| **1 前序** | ⓪ 配置+记忆+模式 | 必须最先完成，模式决定分支 |
| **2 并行** | ① 画像 + ③ 参考数据 | 互不依赖 |
| **3 并行** | ② 清单 + ⑤a 外部信号 | ⑤a 仅依赖 name 列表，可预读 |
| **4 串行** | ④ 评分+冗余+健康 | 依赖②和⑤a结果 |
| **5 并行** | ④-bis 深读 + ⑤b 推荐 | ④-bis 依赖路径(阶段2)，与⑤b无关 |
| **6 串行** | ⑥ 报告 → ⑥-bis 验证 | 依赖全部 |
| **7 串行** | ⑦ 执行 → ⑧ 持久化 | 依赖验证结果 |

**规则：** 同阶段可交换；阶段内失败不阻断其他步骤；失败标记错误在报告标注；`⟳`可并行 `→`串行。

> 实际执行时，阶段2-3-4-5 约减少 35-40% 的 agent tool call 轮次。

### ① 当前项目画像（聚焦扫描 · 多维画像）

> **只扫描当前工作目录**（或用户指定的 `--path`），不混入 `scan_paths` 的技能安装目录。扫描范围精确、画像精准。

**第一步：目录基础信息**
- 项目名称（从目录名或 `README.md` 首行 / `package.json` `name` 字段）
- 项目根路径
- 目录结构概览（`ls` 第一层，识别子目录用途：`tools/`、`scripts/`、`repos/`、`memories/` 等）

**第二步：关键文件深度分析**
Read 以下文件（存在则读，不存在跳过）：
- `README.md` → 提取项目用途、快速命令、目录结构说明
- `AGENTS.md` / `MEMORY.md` → 提取行为约束、环境、技术栈、兴趣领域、关键路径
- `package.json` / `pyproject.toml` / `Cargo.toml` / `go.mod` → 提取依赖和技术栈
- `.gitignore` → 了解项目排除规则

**第三步：文件特征统计（尊重 `.gitignore`）**
- 语言文件分布：`.py`, `.js`, `.ts`, `.rs`, `.go`, `.java`, `.cs`, `.rb`, `.c`, `.cpp` 等计数
- 配置文件：`package.json`, `pyproject.toml`, `Cargo.toml`, `go.mod`, `pom.xml`, `build.gradle`, `Dockerfile`, `.github/` 等
- 测试文件：`*.test.*`, `*.spec.*`, `__tests__/`, `test/`
- 技能相关：`SKILL.md`, `AGENTS.md`, `SKILLS_INDEX.md`
- 脚本工具：`tools/`, `scripts/` 目录下的可执行文件

**第四步：构建多维项目角色画像**

结合步骤⓪加载的项目记忆 + 上述文件分析 + **数据驱动的痛点推导**，输出：

```text
┌─ 项目画像 ──────────────────────────────┐
│ 目录 {path}  类型 {从文件+project-types} │
│ 用途 {一句话}  活动 {5-8项核心活动}       │
│ 技术栈 {语言+框架+工具}  兴趣 {领域}     │
│ 痛点 {数据驱动}  入口 {关键路径}         │
└─────────────────────────────────────────┘
```

**痛点推导规则（v5.2.0 — 数据驱动，禁止 agent 猜测）：**

痛点必须来自以下三个数据源之一，不得凭空推断：

| 数据源 | 导出痛点 |
|--------|---------|
| usage 存在 T1 但 0 次调用 | "技能覆盖率低: {技能} T1核心但0次调用" |
| usage 全集中在单领域 | "技能使用偏科: 集中在{领域}" |
| 大量 T3 仍已安装 | "技能库臃肿: {N}个T3占用扫描路径" |
| config scan_paths 指向不存在目录 | "配置失效: {路径}不存在" |
| 技能依赖的外部工具缺失 | "依赖断裂: {技能}依赖{工具}已缺失" |
| 同类工具多版本共存 | "版本碎片: 多版本共存" |

**痛点输出示例（带数据来源标注）：**

```
痛点   技能覆盖率低: skill-creator(T1/89分) 0次调用 · 建议开启定期审计
       技能库臃肿: 8 个 T3 技能占用扫描路径 · 需评估是否归档（来源: 步骤④-③容量预警）
```

> 无任何数据源命中时，痛点字段**输出空行**（不写任何推断文本）。宁可空白不要"可能的瓶颈"这类废话。

画像示例：
```
类型    AI Agent 技能开发 (ZCode)
用途    AI Agent 辅助编码，开发/调试/投研/游戏/写作/玄学
活动    技能编写 · Agent编排 · 工具链维护 · MCP集成 · 互联网调研 · 游戏制作 · 炒股投研 · 写作
技术栈  Python 3.12 · Node.js v24 · PowerShell · Markdown
兴趣    🎮游戏 · 🔮玄学 · 📈炒股 · ✍️写作 · 📋文案
痛点    技能覆盖率低: skill-creator(T1/89分) 0次调用（来源: usage历史）
入口    tools/learn.py · tools/agent-reach/cli.py
```

> **画像即锚点**：后续所有步骤（分层、评分、搜索、推荐）都以此为唯一尺度。
>
> **模式分支**: `$AUDIT_MODE = "project"` 时，按并行阶段执行阶段1→2→3→4→6，跳过阶段5（④-bis深读 + ⑤b C层搜索）。

### ② 已安装技能清单

**第一阶段：扫描 SKILL.md（原有逻辑）**

扫描 `config.yaml` 中所有 `scan_paths`，提取每个技能的：
- `name`、`description`、路径、来源（user/system）、版本号（从 SKILL.md frontmatter 或 VERSION 文件读取）、可编辑性

**分类**：
- **用户技能**（editable=true，用户可自行增删改）
- **系统/插件技能**（editable=false，来自插件缓存）

**输出:**
```
🔍 已安装技能: 用户 {N} 个 + 系统/插件 {N} 个
```

**去重**：同 `name` 的技能保留一个——优先 editable=true（用户版）→ 同 editable 则选版本号高者。去重数在输出中标注（如"去重 2 个"）。

**第二阶段：扫描 MCP 服务器（v5.3.0 新增 — 跨层能力发现）**

> 技能审计的边界从此从 SKILL.md 扩展到整个能力生态。MCP 工具是"可执行能力"的重要组成部分，不可忽略。

读取 `config.yaml` 中 `mcp_config.files` 路径列表，逐个文件：
1. 定位 `mcp.servers` 块（ZCode 格式）或 `mcpServers` 块（Claude Code 格式）
2. 对每个 MCP 服务器，提取：
   - `name` — 服务器名称（如 `codegraph`、`headroom`）
   - `command` — 启动命令（用于分类：npx/npm = Node.js, python/pip = Python, go = Go, 单二进制 = binary）
   - 从 `mcp_config.tool_catalog` 缓存（如存在）读取工具列表；无缓存时标注"⏸ 工具列表未缓存，仅记录服务器级信息"
3. 对每个服务器，关联其工具列表（名称 + 短描述取前80字符）

**输出格式（追加在技能清单后）：**
```
🔧 MCP 服务器: {N} 个
  {服务器名} → {command 简略} · {N}工具 · {语言/类型}
  ...
```

**总能力统计：**
```
📊 总能力来源: 技能 {N} + MCP 服务器 {M}（{T}工具）
```

### ③ 加载参考数据

从 disk 读取 `references/project-types.yaml`（项目类型→推荐技能映射）和 `references/skill-registry.yaml`（已知技能版本号+分类），加载到内存供步骤④使用。

> 找不到 references 文件时报 warning，继续执行（降级为仅靠内置默认规则+web 搜索推荐）。
> 此步不做任何筛选判定，仅为后续步骤准备数据。

### ④ 三级分层 + 五维评分 + 描述检查

**第一步：三级分层（以项目角色画像的"活动"为尺子）。**

| 层级 | 规则 | 动作 |
|------|------|------|
| **T1 核心** | 技能的主要用途直接命中"核心活动"中的某一项 | 保留（自动） |
| **T2 通用** | 技能不局限于特定项目类型，当前项目可用，无负面影响 | 保留 |
| **T3 专业** | 技能为特定项目类型设计，与"核心活动"零交集 | 建议归档 |

> 判定密钥：把技能的 description 和项目画像的"活动"做交集。有交集 → T1；无交集但通用类别(debug/docs/review/管理/验证/设计等) → T2；明确服务其他类型(Android/iOS/特定迁移) → T3。
> T1 技能之间做功能重叠检测：description 关键词相似度 >60% 则标 ⚠️ 建议二选一（如 subagent-driven-development vs dispatching-parallel-agents）。

**第二步：五维评分（仅已安装技能，每技能 0-100）。Community 维度暂用默认值 50，步骤⑤获取真实数据后回填。ROI 维度基于 SKILL.md 行数和 usage 历史估算。**

| 维度 | 权重 | 计算方式 |
|------|------|---------|
| **Fit** 项目匹配 | 35% | **连续计分制**（消除断崖分档）：= (活动关键词交集数 / 画像核心活动总数) × 100。每命中 1 项 +15 分；同义词通过同义词表匹配（编写≈创作≈开发）。无交集 → 0 分。T1/T2/T3 仅用于健康管理的归档建议，不参与评分。 |
| **Value** 项目价值+描述质量 | 20% | ①类别微调（±5）：debug/验证/审查 +5；文档/笔记/管理 持平；设计/规划 −0；纯平台工具 −5。②使用频率：读取 `usage-*.json`（不存在时所有技能统一 +0，不惩罚），高频调用 +5；低频 −5；零调用 −10。③描述质量分：中文且格式合格 → +10；仅英文 → −10；超长/多行 → −10 |
| **Fresh** 版本时效 | 15% | **降权**。SKILL.md 类技能的版本号语义弱于二进制包。版本 = registry 最新 → 100；落后 1 大版本 → 70；registry 无记录 → 70（本地技能不扣分）；落后 2+ → 50；无版本号 → 60 |
| **Community** 外部信号 | 20% | **改为可选加成**（不惩罚本地技能）。默认 50（而非 25）。步骤⑤回填规则：GitHub ⭐≥1000→100，≥100→85，≥10→70，<10→55；npm 📥≥1000/w→100，≥100/w→80，<100→60；最近更新 1 月内→100，6 月内→80，1 年内→65，>1 年→50。本地开发无公开 repo → 保持 50，不影响"合格"判定。 |
| **ROI** 成本效益比 | 10% | **新增**。= 预估节约 token/次 / (预估消耗 token/次 + 1) × 100。节约/消耗数据来源见步骤⑤b「ROI预估」节。零调用技能默认中性值 50。高频高回报 → 100；低频无回报 → 20。 |

> **ROI 预期行为**：大型技能(>200行)天然 ROI 偏低(≈20-40)，小型偏高。这不是缺陷——ROI 反映真实成本效益比，权重仅10%，核心技能仍可通过高 Fit+Value 合格。

**总分** = Fit×0.35 + Value×0.20 + Fresh×0.15 + Community×0.20 + ROI×0.10
**建议**：≥70 ✅保留 · 40-69 🟡低频监控 · <40 ❌建议归档

> Fit 连续计分，T1/T2/T3 降级为归档参考；Community 默认 50(本地不低分)；Fresh 从 20%→15%

**第三步：冗余检测 + 健康管理（新增 v5.1.0）**

对已安装技能执行以下健康检查：

**① 功能重叠检测**

**技能-技能重叠（原有）：**
- 取所有 T1/T2 技能的 `description` + `trigger_words`，做关键词交集
- 关键词重叠度 > `health_thresholds.overlap_threshold`（默认 60%）→ 标 ⚠️ 建议二选一
- 输出格式：`⚠️ {技能A} ↔ {技能B}  重叠度{NN%} · 共同关键词: {kw1, kw2, ...}`

**技能-MCP 跨层重叠（v5.3.0 新增）：**
- 取已安装技能的 `description` 关键词 vs MCP 服务器的名称+工具描述关键词
- 匹配规则：技能描述中 2+ 关键词命中 MCP 工具描述 → 标记 ⚠️
- 输出格式：`⚠️ {技能名} ↔ MCP/{服务器名}.{工具名}  重叠: {关键词}`
- **特殊规则**：互补关系（如"方法论指引技能"+"执行工具 MCP"）标记 ✅互补，不标 ⚠️
  - 方法型描述（含"方法论/流程/规范/步骤/指南"）+ 执行型描述（含"执行/操作/运行"）→ 判为互补
  - 例: `finishing-a-development-branch(分支流程) ↔ github(PR/merge 执行)` → ✅互补

**② 使用率四象限矩阵**
从 `usage-*.json` 读取调用次数 + 评分映射到象限：高频高值✅ 高频低值🔶 低频高值🟢 低频低值🔴

**③ 容量预警**
- 检查 `config.yaml` 的 `health_thresholds`:
  - 总技能数 > `max_total_skills`（默认 30）→ 🚨 `技能数 {N} 超过阈值 {max}，建议清理 T3 技能`
  - T3 占比 > `max_t3_ratio`（默认 30%）→ 🚨 `T3 占比 {NN%} 超过阈值，建议归档 {list}`
  - 调用次数 = 0 的技能 → 🟡 `0 次调用，评估是否保留`

**④ 依赖链检查**
对每个用户技能，检查其 SKILL.md 中引用的外部命令/文件是否存在：
```
{技能名} → 依赖检查:
  ✅ ffmpeg → C:\Tools\ffmpeg-8.1.1\bin\ffmpeg.exe
  ❌ tesseract → MISSING（可选，不影响核心功能）
```

**第四步：描述质量检查（每个已安装的用户可编辑技能）。**

检查规则与精炼生成步骤详见 `references/description-quality.md`。此处仅列核心规则：

- **语言**：根据 `output_language`（auto/zh/en）决定目标语言
- **格式**：`触发词 → 功能简介` 结构，≤40 字符
- **精炼**：5 步——提取触发词→构建简介→截断→翻译→双语拼接
- **自指**：审计完成后必须检查 skills-audit 自身 description（解决自指盲区）
- **系统技能**：editable=false 只标注不修改

**评分表行格式（已安装技能）：**
```
#N  技能名           分  层级  ⭐外部信号
```

示例：
```
#1  skill-creator    89  T1    内置·官方
#5  sub-agents       78  T1    ⭐44·6月新  跨LLM子代理
```


### ④-bis 逐技能深度阅读与分析（新增 v5.0.0）

> **这是整个审计的核心价值所在。** 不读技能正文的审计 = 盲评。

对每个已安装用户技能（editable=true），执行以下深度分析：

**第一步：读全量内容**
- Read 技能 SKILL.md 完整文件（不限行数）
- 提取：核心能力清单、触发词、依赖项、引用的外部文件/路径、输出格式定义

**第二步：逐技能六维分析**

> 详细输出模板（含六维分析/维护清单/参考验证格式）已外移至 `references/deep-analysis-template.md`。

对每个技能，按此流程执行后，引用模板格式输出：
1. 读正文 → 2. 分析作用·适配·缺口·建议 → 3. 维护清单（版本/环境/描述/注意）→ 4. 外部参考验证（agent-reach/web_fetch ≥2来源）→ 5. 自指校验
- 缺口必须经工具验证（ls/Test-Path/grep）
- 网络不可用时标注「⏸ 离线，建议网络恢复后验证」

**第五步：自指校验**
- 检查 skills-audit 自身的深度分析是否被应用（解决自指盲区）
- 若用户执行了建议，下次审计时回测建议效果


### ⑤a 外部信号获取（缓存优先）

从 `external-signals-cache.json` 加载缓存（首次运行为空）。

对每个已安装且 registry 中有 homepage 的技能，判定：
- **有缓存且未过期** → 直接用缓存值（GitHub ⭐ 缓存 7 天，npm 📥 缓存 30 天）
- **无缓存或已过期** → WebFetch 获取后写入缓存，标注 `fetched_at` 时间戳
- **网络不可用** → 用缓存值（忽略过期），无缓存则用默认值 25

```
GitHub: WebFetch api.github.com/repos/{owner}/{repo}
        → stargazers_count, pushed_at, description
npm:    WebFetch api.npmjs.org/{package}/latest
        → version, weekly downloads
PyPI:   WebFetch pypi.org/pypi/{package}/json
        → info.version, downloads
```

> 同一 GitHub org 下多个 repo 合并为单次 API 请求。缓存写入 `external-signals-cache.json`。
> 首次运行全量获取较慢，后续仅增量获取新技能或过期条目，大幅提速。
> 获取完成后将真实 Community 分回填到步骤④的评分表。

### ⑤b 三层推荐引擎（画像驱动 · 置信度量化）

> **黄金法则**：每一条推荐都必须附带信心指数（0-10）和一组可追溯的证据链。禁止空口推荐。
> 信心指数 < 5 的推荐自动折叠到"仅供参考"区，不占主推荐位。

---

**置信度评分规则（全局统一）：**

| 维度 | 权重 | 计算方式 |
|------|------|---------|
| **匹配度** — 技能与画像活动的语义覆盖 | 40% | 活动交集数 / 画像总活动数 × 10；同义词组（编写↔创作↔开发）使用 config.yaml 的同义词映射表 |
| **数据源** — 推荐来源的可靠等级 | 30% | project-types.yaml 手动维护 → 9；skill-registry.yaml 注册 → 7；Web 搜索 ≥2 来源一致 → 6；Web 单来源 → 4；agent 推测 → 2 |
| **交叉验证** — 多个独立来源的一致性 | 20% | ≥3 来源一致 → 10；2 来源一致 → 8；单一来源 → 5；来源矛盾 → 0（并在报告中标注冲突） |
| **时效性** — 推荐数据的更新时间 | 10% | 30 天内 → 10；6 个月内 → 8；1 年内 → 6；>1 年 → 3；未知 → 5 |

**综合信心指数 = 匹配度×0.4 + 数据源×0.3 + 交叉验证×0.2 + 时效性×0.1**

---

**ROI（投资回报率）预估（每项推荐必算）：**

```
ROI = 预估节约 token/次 − 预估消耗 token/次
```

| 参考基准 | tokens/run | 来源 |
|---------|-----------|------|
| SKILL.md <50行 | ~500 | 全文行数估算 |
| 50-200行 | ~1,200 | 同上 |
| >200行 | ~2,500 | 同上 |
| 仅description | ~200 | 最小加载成本 |
| 高频任务(调试/搜索/总结) | 节约+1,000~3,000 | 替代人工轮数 |
| 中频任务(审查/验证) | 节约+300~1,000 | 结构性辅助 |
| 低频/领域工具 | 净消耗-500 | 无直接效率提升 |

> ROI > 0 → 推荐安装；ROI < −500 且非核心需求 → 标注"高成本低回报，谨慎评估"。

---

**第一层（A层）：内置映射匹配 + 置信度**

从 `references/project-types.yaml` 匹配当前画像.类型，对每条推荐计算匹配度和置信度。

> 完整示例见 `references/recommendation-examples.md#a层内置映射`。

**输出格式（每行必填）：**
```
{等级} {技能名}  信心:{分}/10  ROI:{分}/run
  📌 命中: {活动覆盖说明}
  📊 交叉验证: {来源说明}
  💡 预期效果/决策建议: {...}
  ⚠️ 注意/冲突检测: {...}
```

> 未匹配到类型时跳过此层，继续 B 层。

---

**第二层（B层）：Gap 分析 — 未覆盖活动 → 方案评估**

1. 从画像.活动 + 画像.兴趣 提取所有活动关键词
2. 与已安装技能 description 做交集
3. 未覆盖活动 → 评估相关性 + 给出决策建议（而非单纯标记缺失）

> 完整示例见 `references/recommendation-examples.md#b层gap-分析`。

**输出格式：** 覆盖分析表（技能名 ✅/🔴）+ 未覆盖活动评估（相关性·场景判定·建议·风险提示）

---

**第三层（C层）：网络搜索补充 + 评估矩阵**

搜索执行规则：
- 中英文各 1 次搜索（各取前 5 条结果）
- 对每项结果计算信心指数（匹配度 + 数据源 + 交叉验证 + 时效性）
- 同名/同功能技能去重：保留信心指数最高的来源
- 与已安装技能冲突时：对比信心指数，新结果更高才展示
- 外部技能描述同样执行描述质量检查

> 完整示例见 `references/recommendation-examples.md#c层网络搜索评估`。

**输出格式：** 评估矩阵表 + 前 3 条详细评估（活动覆盖·来源等级·交叉验证·冲突检测·安装命令）

---

**推荐终选输出规则：**
- A层 + C层 合并排序，按信心指数降序
- 信心 ≥ 7 → ⭐⭐⭐⭐ "强烈推荐" → 主推荐位
- 信心 5-6.9 → ⭐⭐⭐ "建议评估" → 主推荐位（折叠展示）
- 信心 < 5 → ⭐⭐ "仅供参考" → 自动移入"不推荐"区
- 同技能多来源时取最高信心的来源
- 已安装技能不进推荐表（通过覆盖分析标注✅）

**不推荐清单规则（自动化判定）：**

| 来源 | 判定条件 | 输出格式 |
|------|---------|---------|
| A层/C层信心<5 | 4 个维度加权平均 < 5（匹配度+数据源+交叉验证+时效性） | ❌ {技能} 信心:{分}/10 ROI:{分}/run |
| A层/C层 ROI < −1,000 | 净消耗过大且无补偿性核心价值 | 同上 + 标注"高成本低回报" |
| 已安装 T3 且 < 50分 | 零交集 + 低分 | ❌ {技能} T3/{分} 零交集 |
| 已安装 0 调用 > 30 天 | usage-*.json 无记录 | ❌ {技能} 0次/30d 建议卸载释放空间 |
| 已安装且与核心技能重叠 > 70% | 功能重叠检测结果 | ❌ {技能} 与{其他}重叠{NN%} 考虑合并 |

> 不推荐清单必须具备可追溯的判定原因（基于数据而非 agent 感受）。
> 没有数据支撑的不推荐条目禁止输出。

> 完整示例见 `references/recommendation-examples.md#不推荐清单格式`。

**不推荐清单格式：**
```
❌ {技能}  {原因}
  ⚠️ {详细判定依据（引用数据源）}
  📊 {量化证据（调用次数/分数/成本等）}
  💡 {替代方案建议}
```

> **核心原则**：每块必须在"有实质内容"时才输出。无内容时**不输出任何兜底文本**。典型报告 3-5 块，不再浪费 token 展示空区。
> **模式过滤**: $AUDIT_MODE = "project" 时，跳过 📋深度分析 + 📝描述优化。

---

**输出规则矩阵：**

| 区块 | 条件 | 行为 |
|------|------|------|
| 📋 **项目角色画像** | 始终 | 必定输出 |
| 📊 **已安装技能评分** | 始终 | 必定输出，按分降序，每条含外部信号 + 健康标记 |

---

**报告模板 → 已外移至 `references/report-template.md`（步骤⑥-bis 验证通过后，agent 按此模板组装最终报告）。**

> 完整模板含 7 个条件区块的 ASCII art 格式。此处仅保留核心格式规则：
> - 📋 画像: 8 字段固定模板
> - 📊 评分表: `#N  技能名  分  层级  ⭐/📥  ROI/run  标记`
> - 🔧 MCP 能力地图: 按功能域分组（🟢独家 · 🟡与技能重叠 · 🔵能力缺口 · ✅互补），格式:

>     🟡 MCP/{server}.{tool} ⚠️ 与{skill}重叠
>     🔵 (缺口) — {缺失能力}
>   ```
> - 📦 推荐: `推荐（信心≥7）` + `不推荐（信心<5或ROI<-1,000）`

> - 📝 描述优化: `{技能} 问题 → 建议`

### ⑥-bis 审计结果验证（v5.2.0 新增 — 确保报告可信）

> 审计报告生成后，在执行任何操作之前，先验证审计结果自身的可信度。**这是对审计员的审计。**

**第一步：数据一致性检查**

| 检查项 | 方法 | 失败处理 |
|--------|------|---------|
| 评分表与画像一致 | 确认评分表的 Fit 维度最高分技能确实命中了核心活动 | 不一致 → 在报告末尾标注⚠️"评分-画像偏差" |
| Community 回填状态 | 确认 `external-signals-cache.json` 在步骤⑤a之后已写入 | 未回填 → 标注"部分信号未获取，评级仅供参考" |
| 推荐与不推荐无冲突 | 确认同一技能不同时出现在推荐和不推荐两大清单中 | 立即更正，标注⚠️"已自动修正冲突" |
| 去重完整性 | 确认评分表中无同名技能 | 存在重复 → 标注⚠️"去重遗漏，建议手动检查" |

**第二步：抽样交叉验证（轻量级，仅 1-2 个技能）**

随机选取 1 个 T1 技能和 1 个 T3 技能，验证其评分合理性：

```
抽样验证:
  [T1] skills-audit 评分55分
    ├ 画像活动: 技能编写·Agent编排·工具链维护·MCP集成·互联网调研
    ├ Fit=40(2/5项交集) → 与 description 比对: "技能审查"涉及编写环节 ✅
    └ 合理性判定: ✅ 通过（核心工具但消耗大→低ROI拉低总分）
  [T3] android-dev 评分36分
    ├ 活动交集: 0/5项 → Fit=0
    └ 合理性判定: ✅ 通过（零交集，建议归档）
```

> 抽样验证不需要用户参与，agent 自行完成比对判断即可。
> 验证发现异常时在报告末尾追加"⚠️ 验证提醒"段落，**不阻塞执行流程**。

**第三步：验证结果输出**

验证完成后，在报告末尾追加：

```
✅ 审计验证: 通过
  - 数据一致性: 4/4 项检查通过
  - 抽样验证: T1 合理 · T3 合理
  - 报告可信度: 高（所有评分可追溯到活动交集计数）
```

### ⑦ 执行（用户确认后）

**操作范围**：仅用户层技能（editable=true）。系统/插件层技能（editable=false）只读——可评分、标注、建议，禁止任何 Edit/Write/Bash 修改操作。

- **归档**：技能目录**复制**到 `config.yaml` 的 `archive_dir`，原路径重命名为 `{原名}.archived`（非删除，可手动恢复）
- **修改描述**：① 同目录创建 `.description.bak` 备份文件 ② Edit 对应技能 SKILL.md frontmatter 的 `description:` 行，替换为优化版描述
- **安装**：给出安装命令示例（clone / 下载 / 安装脚本）
- **更新**：给出更新方式
- **搜索安装**：给出来源 URL
- **回滚**：报告末尾附回滚命令——`还原描述: cp/mv (Linux) 或 copy/move (PowerShell)` / `还原归档: mv {name}.archived → {name} (Linux) 或 move (PowerShell)`

### ⑧ 持久化（始终执行）

1. 项目画像 → 双写：① `{project}/.agents/project-profile.md`（人类可读，与项目记忆同目录，头部标注 `关联: L0=~/.agents/AGENTS.md, L2=memories/MEMORY.md`）② `project-profiles.json`（以项目 hash 为 key，含阶段+技能使用历史字段，覆盖不追加）
2. 运行统计 → `stats.json`（仅存汇总统计，不含每条明细）
3. 审计日志 → `activity-log.jsonl`（保留最近 50 条，超出自动截断。每条记录含本次审计评分+各技能调用次数（从 `usage-*.json` 读取））
4. 外部信号缓存 → `external-signals-cache.json`（以技能 name 为 key，含 fetched_at 时间戳。仅供下次运行增量获取用）
5. 审计快照 → 上次审计数据从 `activity-log.jsonl` 倒数第 2 条读取，写入报告生成 `📈 对比上次审计` 行

## 项目类型识别规则

详见 `references/project-types.yaml`。核心推断规则：

- **主特征定类型**: `package.json`(React/Vue/Angular/Next/Express) → Web前端/Node后端/全栈；`Cargo.toml` → Rust；`pyproject.toml` → Python；`go.mod` → Go；`pom.xml`/`build.gradle` → Java；`.csproj` → C#；`pubspec.yaml` → Flutter；`CMakeLists.txt` → C/C++；`Gemfile` → Ruby
- **AI Agent 增强**: 多 `SKILL.md` + `MEMORY.md` → `⭐AI Agent 技能开发`
- **辅助识别**: `requirements.txt` 含 pandas/numpy → 数据科学；`Dockerfile` → 容器化
- **多特征叠加**: 多类型共存时合并（如 react+express → "全栈: React+Express"）

## 边界规则（快速参考，详情见各步骤正文）
